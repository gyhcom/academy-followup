# 200명 운영형 데이터 모바일 파일럿 UAT 결과

## 1. 검수 목적

친구 학원 파일럿 전에 200명 운영형 데이터 기준으로 `홈 -> 출석 -> 문자 -> 관리` 흐름이 모바일 390px에서 실제 운영 가능한지 확인했습니다.

이번 검수는 수정 작업이 아니라 제품화 판단을 위한 UAT입니다. 코드, API, DB schema, migration, seed 파일은 변경하지 않았고, 발견한 문제는 후속 GitHub Issue 후보로만 정리했습니다.

## 2. 검수 환경

- 기준 브랜치: `codex/t-612-200-student-mobile-uat`
- 기준 커밋: 최신 `origin/main`에서 분기
- 기준 URL: `https://academy-followup.vercel.app/app`
- 기준 데이터: Production Supabase의 더배움프라임영수학원 운영형 테스트 데이터
- 기준 화면 폭: 모바일 390px, 높이 852px
- 보조 확인: Playwright headless Chromium
- 테스트 계정:
  - 원장: `owner@test.com / 1234`
  - 선생님: `teacher@test.com / 12345678`
  - 보조 선생님: `assistant@test.com / 12345678`
- `sms_dry_run` 상태: 실제 발송은 실행하지 않았고, dry-run 전제의 화면/권한/차단 흐름만 확인
- 검증 명령:
  - `npm run lint`: 통과
  - `npx tsc --noEmit`: 통과
  - `npm run build`: 통과
  - `git diff --check`: 통과

주의: 이 UAT를 처음 작성한 검수 시점에는 200명 seed 문서의 기대값과 Production 화면 값이 달랐습니다. 당시 Production 화면과 리포트 API에서는 재원 학생 172명, 반 19개, 스케줄 미등록 74명으로 확인되었습니다. 이후 2026-06-03에 Supabase SQL Editor에서 200명 seed 재실행/정리/verify를 완료했고, 현재 파일럿 기준값은 학생 200명, 반 20개, 정규 스케줄 580개, 스케줄 미등록 0명입니다. 따라서 이 문서의 상세 UAT 결과는 과거 검수 시점의 발견 이슈로 보존하고, 이후 검수는 정상화된 200명 기준으로 이어갑니다.

## 3. owner 계정 검수 결과

### 홈

- 로그인 후 첫 화면은 홈으로 진입했습니다.
- 모바일 390px에서 가로 overflow는 없었습니다.
- 홈 상단에 `확인할 학생 31명`, `연락 전 31명`, `오늘 학원 일정 6개 · 보강 불가 2건`이 표시되었습니다.
- 오늘 일정은 `학원 일정`과 `공유된 타 학원 일정`으로 분리되어 표시되었습니다.
- 공유 일정은 `공유됨`, `연결 학원 수업`, `학원명 비공개`, `보강 제외`로 표시되어 실제 학원명과 전화번호는 노출되지 않았습니다.
- 학원 일정 row의 출석 이동은 아이콘 버튼 형태로 노출되어 모바일 공간을 덜 차지했습니다.

### 출석

- 출석 탭에서 날짜, 수업 수, 학생 수, 미체크 수, 연락 필요 수가 표시되었습니다.
- 2026.06.01 기준 `수업 6개`, `학생 34명`, `미체크 32명`, `연락 필요 1명`이 확인되었습니다.
- 수업 리스트는 시간순으로 보이고, 선택 수업의 학생 목록에서 `도착`, `지각`, `예외` 액션이 확인되었습니다.
- 첫 선택 수업은 8명 규모였고, 모바일 한 화면에서 여러 학생 row를 연속 확인할 수 있었습니다.
- 실제 출석 상태 변경은 데이터 변형을 피하기 위해 실행하지 않았습니다.

### 문자

- 문자 탭에는 `개별 연락 / 전체문자` 전환이 표시되었습니다.
- 개별 연락은 반 선택 후 학생 목록과 사유 버튼이 표시되었습니다.
- 원장 계정에서는 학부모 연락처가 마스킹된 숫자로 표시되었습니다.
- 전체문자 화면에는 대상 범위 `전체 학생 / 반 선택 / 학년 선택`, 수신자 `학부모 / 학생 / 학부모+학생`, `중복 수신자 제외` 안내, byte/SMS 표시가 확인되었습니다.
- 전체문자 본문은 `44자 · 77byte · SMS 예상`으로 표시되었습니다.
- 실제 dry-run 발송은 데이터 변경을 피하기 위해 실행하지 않았습니다.

### 관리

- 관리 탭은 `시작 / 명단 / 수업 / 직원 / 문자 / 정책 / 리포트 / 이력` 메뉴 구조로 표시되었습니다.
- 관리 상단 요약은 `학생 172명`, `미등록 74명`, `반 19개`, `직원 5명`으로 표시되었습니다.
- 원장 시작 순서와 CTA는 모바일에서 접근 가능했습니다.
- `명단`, `수업`, `직원`, `문자`, `정책`, `리포트`, `이력`이 업무 단위로 분리되어 원장용 정보 구조는 이전보다 제품형 관리 콘솔에 가깝습니다.

### 리포트

- `/api/reports/summary?range=7d`는 200으로 응답했습니다.
- 최근 7일 기준 요약:
  - 출석 총 194건
  - 출석 46, 지각 8, 결석 3, 확인 필요 5, 보강 3, 미체크 129
  - followup 5건, message log 5건
  - dry-run 5건, 실발송 0건, 실패 0건
  - 학생 172명, 반 19개, 스케줄 미등록 74명
  - audit log 1건
- 학생 CSV 다운로드 API는 200으로 응답했고 기본값에서는 전화번호가 마스킹되었습니다.
- 원장 계정에서 `includePrivate=true` 요청 시 원문 전화번호가 포함된 CSV가 반환되었습니다.

### 이력

- 최근 변경 이력 1건이 표시되었습니다.
- 감사 로그 CSV API는 200으로 응답했습니다.
- 감사 로그 sample에는 학생 전화번호 원문이 포함되지 않았고, `UAT 고예린 학생 정보를 수정했습니다.` 같은 요약만 포함되었습니다.

### 발견 이슈

- Production 데이터가 200명 seed 기대값과 다릅니다.
- 전체문자 발송 전에 `대상 N명 / 실제 발송 M건 / 중복 제외 K건`이 바로 보이지 않아 원장 확인성이 약합니다.
- 스케줄 미등록 학생이 74명으로 많아 관리 첫 화면에서 파일럿 준비가 덜 된 것처럼 보입니다.

## 4. teacher 계정 검수 결과

### 담당 반 필터

- 선생님 계정은 하단 탭에서 `홈 / 문자 / 출석`만 표시되고 `관리` 탭은 숨겨졌습니다.
- 문자 탭에서는 담당 반 1개, 학생 10명만 표시되었습니다.
- 전화번호 숫자 대신 `학부모 연락처 등록됨`으로 표시되어 전화번호 비노출 정책이 유지되었습니다.

### 출석

- 2026.06.01 기준 출석 탭에는 `수업 0개`, `학생 0명`으로 표시되었습니다.
- 담당 반이 문자 탭에는 보이지만, 해당 날짜 출석에는 수업이 없어 빈 상태가 표시되었습니다.
- 빈 상태 문구는 표시되지만, 홈 상단의 전체 학원 일정 요약과 충돌해 선생님 입장에서는 “오늘 수업이 있는지 없는지” 판단이 헷갈릴 수 있습니다.

### 문자

- 담당 반 학생 10명이 표시되었습니다.
- 학생별 `문자 작성`, `결석`, `재시험`, `숙제 미완료` 버튼이 표시되었습니다.
- 학부모 전화번호 원문 또는 마스킹 숫자는 노출되지 않았습니다.
- 보강 달력과 주간 스케줄 영역은 표시되었습니다.

### 관리 탭 제한

- 관리 탭은 UI에서 숨겨졌습니다.
- `/api/reports/summary?range=7d` 직접 접근은 403으로 차단되었습니다.

### 전화번호 비노출

- 문자 화면에서 `학부모 연락처 등록됨`만 표시되고 전화번호 숫자는 노출되지 않았습니다.

### 발견 이슈

- 홈 상단은 전체 학원 기준 `오늘 학원 일정 6개 · 보강 불가 2건`을 보여주지만, 선생님 상세 일정은 비어 있어 역할별 정보 위계가 어긋납니다.
- 출석 탭에서 담당 반이 없는 날짜에는 정상 빈 상태지만, “담당 반이 있는 다른 날짜 보기” 같은 빠른 안내가 없어 테스트/운영 중 혼란이 있을 수 있습니다.

### 팔로업

- 문자 탭에서 학생별 사유 버튼과 보강 달력/최근 팔로업 영역은 접근 가능했습니다.

### 관리 탭 제한

- 관리 탭은 UI에서 숨겨졌습니다.
- `/api/reports/summary?range=7d` 직접 접근은 403으로 차단되었습니다.

### 발견 이슈

- 보조 선생님 실제 발송 제한은 버튼 클릭까지 포함한 별도 쓰기 테스트가 필요합니다.

## 6. 모바일 390px UX 결과

### 가로 overflow

- owner, teacher, assistant 주요 화면에서 `scrollWidth === clientWidth`로 확인되어 가로 overflow는 없었습니다.

### 하단 탭

- owner: `홈 / 문자 / 출석 / 관리`
- teacher: `홈 / 문자 / 출석`
- assistant: `홈 / 문자 / 출석`
- 권한별 탭 노출은 의도대로 동작했습니다.

### 바텀시트

- 이번 자동 검수에서는 학생 상세/문자 작성 바텀시트의 실제 열림까지는 확인하지 않았습니다.
- 문자 탭에서 학생 선택과 하단/보조 작성 흐름으로 이어질 수 있는 버튼은 확인되었습니다.

### 긴 리스트

- 원장 문자 탭에서 반 14개, 학생 100명으로 표시되었습니다.
- 반 선택 칩/버튼이 길게 이어지고, 200명 전체 데이터 기준에서는 학생 검색/필터가 없는 구간에서 스크롤 피로도가 생길 수 있습니다.
- 관리 탭은 메뉴 그룹이 분리되어 첫 화면은 이해하기 쉬우나, 명단 172명 규모에서 실제 학생 검색/수정까지는 별도 수동 검수가 필요합니다.

### 검색/필터

- 문자 개별 연락은 반 필터 중심입니다.
- 전체문자는 대상 범위 필터가 있습니다.
- 관리 명단의 검색/필터는 이번 자동 검수에서 상세 조작까지 확인하지 않았습니다.

### 스크롤 피로도

- 홈은 오늘 일정과 공유 일정이 compact하게 정리되어 첫 화면 부담은 낮습니다.
- 출석은 한 수업 8명 기준에서는 사용 가능하지만, 15명 이상 수업에서 실제 토글 속도와 저장 체감은 별도 확인이 필요합니다.
- 문자 탭은 반/학생 리스트, 보강 달력, 최근 팔로업이 한 흐름에 있어 장시간 사용 시 스크롤 피로도가 남을 수 있습니다.

## 7. 200명 데이터 기준 성능/사용성 결과

### 학생 검색

- Production 기준 재원 학생 수는 172명으로 표시되었습니다.
- 200명 기준 검수 전 실제 seed 적용 여부를 먼저 맞춰야 합니다.
- 관리 명단에서 학생 검색/수정이 실제로 빠른지는 이번 자동 검수만으로 결론 내리기 어렵습니다.

### 반 필터

- 원장 문자 탭에서는 반 14개가 표시되었습니다.
- 선생님/보조 선생님은 담당 반 1개만 표시되어 권한 필터는 정상으로 보입니다.

### 출석 체크

- 원장 기준 수업 6개, 학생 34명, 미체크 32명이 표시되었습니다.
- 선택 수업 8명 기준으로 `도착 / 지각 / 예외` 버튼이 한 줄 흐름으로 확인되었습니다.
- 실제 상태 저장은 이번 문서 검수에서는 실행하지 않았습니다.

### 전체문자 대상 선택

- 전체문자 화면에서 전체 학생/반/학년 범위와 수신자 선택은 확인되었습니다.
- 발송 전 대상 수와 중복 제외 수가 바로 드러나지 않아 200명 기준 원장 확인성이 부족합니다.

### 관리 명단 탐색

- 관리 첫 화면의 정보 구조는 B2B 관리 콘솔처럼 정리되어 있습니다.
- 다만 스케줄 미등록 74명이 표시되어 파일럿 데이터 준비 상태가 불완전하게 보입니다.

## 8. 발견한 문제 목록

### 문제 1. Production 200명 seed 기대값과 실제 데이터 수 불일치

- 화면: 관리, 리포트
- 계정: owner
- 재현 단계:
  1. `owner@test.com`으로 로그인
  2. 관리 탭 진입
  3. 상단 요약과 리포트 API 확인
- 영향도: 200명 기준 UAT의 신뢰도가 낮아짐
- 우선순위: P0
- 추천 조치: seed 실행 상태와 cleanup 여부를 확인하고, 200명 seed 기대값 또는 실제 Production 데이터 기준을 하나로 고정
