# 파일럿 운영 데이터 UAT 결과 - 2026-05-14

## 목적

친구 학원이 실제로 1주일 써볼 때 막힐 수 있는 지점을 찾기 위해 `홈 -> 출석 -> 문자 -> 관리` 흐름을 운영 데이터 기준으로 점검했습니다.

이번 점검은 실제 SMS 발송 없이 `sms_dry_run=true` 전제로 진행했습니다.

## 점검 환경

- 대상 URL: `http://localhost:3000`
- 계정: `owner@test.com / 1234`
- 화면 폭:
  - 모바일 390px
  - iPad 768px
  - PC 1440px
- DB 상태:
  - 로컬에 `psql`/`supabase` CLI가 없어 seed SQL을 직접 실행하지 못했습니다.
  - 연결된 Supabase DB에는 기존 UAT/볼륨/운영형 데이터가 섞여 있어 관리 기준 학생 171명, 반 18개가 표시됐습니다.

## 발견 및 조치

- 로그인 성공 후 Next router 이동이 자동 브라우저에서 안정적이지 않아 `window.location.assign("/app")`으로 변경했습니다.
- 문자 화면에 `0명` 반이 여러 개 노출되면 원장이 반 선택에서 헷갈릴 수 있어, 연락 대상 화면에서는 학생이 있는 반만 표시하게 했습니다.
- 하단 탭 버튼의 접근성 이름을 `홈`, `문자 보내기`, `출석부`, `관리`로 고정해 빠른 이동 카드와 구분되게 했습니다.
- 모바일 홈의 교육 아이콘 이미지 크기 경고를 제거했습니다.

## 남은 확인

1. Supabase SQL Editor에서 최신 `supabase/seed-demo-operations.sql`을 전체 실행합니다.
2. 기존 UAT/볼륨 데이터가 섞이지 않은 상태에서 반 8개, 학생 64명 기준으로 다시 확인합니다.
3. 실제 학원 CSV 30명 이상을 넣고 문자/출석/관리 화면 밀도와 필터 동작을 다시 확인합니다.
4. 실제 SOLAPI 발송은 아직 켜지 않고 dry-run 로그와 중복 방지까지만 검증합니다.
