# 모바일 UX/버튼/문구 점검 보고서

## 1. 점검 목적

Academy Follow-up을 친구 학원 파일럿에서 실제로 쓰기 전, 모바일 390px 기준으로 버튼, CTA, 문구, 상태 표현을 점검했습니다.

이번 점검의 제품 기준은 “예쁜 SaaS 대시보드”가 아니라 “학원 운영 현황판 + 수업 직후 빠른 처리 도구”입니다. 따라서 아래 질문을 중심으로 봤습니다.

- 원장은 오늘 먼저 처리할 일을 바로 알 수 있는가?
- 선생님은 수업 직후 5~10초 안에 출석 체크를 끝낼 수 있는가?
- 문자/출석/보강/관리 버튼이 행동 중심으로 보이는가?
- 위험하거나 실제 발송에 가까운 동작은 충분히 구분되는가?
- 개발자식 표현이 운영자 화면에 남아 있지 않은가?

검수 기준은 최신 `main`과 `docs/testing/pilot-mobile-uat-200-students-2026-06.md`입니다. 코드, API, DB schema, seed는 수정하지 않았습니다.

## 2. 화면별 UX 문제

### 홈

좋은 점:

- 홈이 `오늘 일정`, `공유된 타 학원 일정`, `빠른 시작`, `오늘 요약`으로 나뉘어 있어 원장용 운영 현황판의 방향은 맞습니다.
- `학원 일정`과 `공유된 타 학원 일정`이 분리되어 출석 대상과 보강 제외 참고 일정이 구분됩니다.
- 공유 일정에서 상대 학원명과 전화번호를 숨기는 정책은 화면에 반영되어 있습니다.

문제:

- `확인할 학생`, `연락 전`, `확인 필요`, `체크 필요`, `미체크`가 서로 가까운 의미로 쓰여 원장이 “오늘 꼭 해야 하는 일”을 한 단어로 이해하기 어렵습니다.
- teacher/assistant 홈에서 전체 학원 기준 요약은 보이지만 담당 상세 일정은 비어 있을 수 있어 역할별 홈의 의미가 약합니다. 이 문제는 별도 이슈 `T-618`로 이미 분리되었습니다.
- `오늘 학원 일정 6개 · 보강 불가 2건`과 `수업 6 · 공유일정 2`처럼 같은 정보를 다르게 표현하는 구간이 있습니다.
- `목록 / 대상 학생 보기` CTA는 무엇의 목록인지 즉시 이해하기 어렵습니다.
- 날짜 컨트롤은 현재 과하지 않지만, 좌우 화살표가 “운영 기준 날짜 변경”이라는 의미를 모르는 사용자는 처음에 헷갈릴 수 있습니다.

추천 방향:

- 홈의 주요 상태는 `미처리`, `연락 필요`, `연락 완료` 중심으로 통일합니다.
- 출석 전 상태는 `체크 필요`, 출석 후 후속 연락 대상은 `연락 필요`로 분리합니다.
- `목록` CTA는 `미처리 보기` 또는 `학생 목록 보기`처럼 대상을 드러냅니다.

### 출석

좋은 점:

- 수업 선택 후 학생 row에서 `도착`, `지각`, `예외`가 한 줄 액션으로 정리되어 수업 직후 빠른 조작 방향은 좋습니다.
- `도착`은 기본 체크, `지각`은 빠른 체크, `예외`는 결석/확인 필요/보강으로 분리되어 원장 요구와 맞습니다.
- `연락 필요만` 필터가 있어 결석/지각 등 후속 문자 흐름과 연결됩니다.

문제:

- `미체크`, `체크 필요`, `확인 필요`가 모두 출석 화면에 등장합니다. 선생님 입장에서는 `미체크`와 `확인 필요`가 어떻게 다른지 헷갈릴 수 있습니다.
- `예외` 버튼은 짧아서 좋지만, 처음 쓰는 선생님은 그 안에 `결석/확인 필요/보강`이 있다는 것을 모를 수 있습니다.
- 결석/지각 후 문자 패널 문구에 `팔로업 기록`, `dry-run`, `초안` 같은 내부 용어가 섞여 있습니다.
- 저장 성공 후 `저장됨`과 `연락 완료` 사이의 상태 차이가 화면 맥락상 명확하지 않을 수 있습니다.

추천 방향:

- 출석 미처리 상태는 `체크 필요`로 통일하고, 이미 결석/지각이 확정되어 연락해야 하는 대상은 `연락 필요`로 표시합니다.
- `확인 필요`는 결석 확정 전 보류 상태로만 사용합니다.
- 출석 문자 패널의 버튼은 `기록 저장`, `테스트 발송`, `발송 완료`로 통일합니다.
- 사용자 화면에서는 `팔로업` 대신 `연락 기록`을 씁니다.
- 원장/관리자용 달력형 출석부는 PC에서는 월간 grid를 기본으로 두되, 모바일 390px에서는 월 전체 grid보다 `월 선택 → 날짜 리스트 → 선택 날짜 상세 리스트` 흐름으로 단순화합니다.
- 선생님/보조 선생님 모바일 출석부는 기존처럼 수업 직후 `오늘 처리` 중심으로 유지합니다.

### 문자

좋은 점:

- `개별 연락 / 전체문자` 전환은 명확합니다.
- 전체문자에서 `중복 수신자 제외` 개념을 설명하고 있습니다.
- byte/SMS/LMS 표시와 2000byte 초과 차단은 실제 발송 전 안전장치로 가치가 있습니다.

문제:

- 화면 일부에 `dry-run 전체문자`, `dry-run 발송`, `팔로업 기록` 같은 개발/운영 내부 표현이 남아 있습니다.
- `문자 발송 테스트`와 `테스트 발송`이 혼용될 가능성이 있습니다.
- 전체문자 화면은 발송 전 대상/실제 발송/중복 제외 count가 약합니다. 이 문제는 별도 이슈 `T-619`로 이미 분리되었습니다.
- `문자 작성`, `문자 초안`, `팔로업 기록 저장`, `문자 발송 테스트`가 단계적으로 이어지지만 초보 원장에게는 “언제 실제로 보내지는지”가 부담스러울 수 있습니다.
- `전체문자 dry-run 발송`은 파일럿 원장에게 특히 부자연스럽습니다.

추천 방향:

- 파일럿 환경에서는 `테스트 발송`으로 통일하고, 설명에만 “실제 문자는 보내지 않고 기록만 남습니다.”를 씁니다.
- `팔로업 기록`은 `연락 기록`으로 바꿉니다.
- 저장 버튼은 `기록 저장`, 발송 테스트 버튼은 `테스트 발송`, 성공 상태는 `테스트 발송 완료`로 통일합니다.
- 전체문자는 `발송 전 확인` 영역을 명확히 둡니다.

### 관리

좋은 점:

- `시작 / 명단 / 수업 / 직원 / 문자 / 정책 / 리포트 / 이력` 구조는 B2B 관리 콘솔로 이해하기 쉽습니다.
- 원장 시작 순서가 있어 초기 세팅 흐름이 보입니다.
- 리포트/이력은 파일럿 운영 신뢰감을 높입니다.

문제:

- `정책` 카드의 `테스트`, `보조 발송 제한`, `dry-run` 의미가 원장에게 직관적이지 않을 수 있습니다.
- 버튼에 `저장`만 있는 폼이 많습니다. 어떤 정보를 저장하는지 모달/섹션별로는 보이지만, 모바일에서는 버튼만 남으면 맥락을 잃을 수 있습니다.
- `비활성`은 구성원/학생/스케줄에서 위험도가 있는 상태인데, 단순 선택값으로 보이면 실수 가능성이 있습니다.
- 리포트 CSV의 `원문 포함`은 개인정보 위험도가 높은데, 모바일에서 경고 문구와 버튼 위계가 충분히 강해야 합니다.

추천 방향:

- 정책 영역은 `실제 문자 발송 막기`를 `테스트 모드`로, 상태값은 `테스트 모드 켜짐/꺼짐`으로 표현합니다.
- 저장 버튼은 가능하면 `학생 저장`, `반 저장`, `설정 저장`, `템플릿 저장`처럼 대상 포함형으로 통일합니다.
- `비활성`은 destructive 또는 warning 스타일과 확인 문구를 붙입니다.
- CSV 원문 포함은 `개인정보 포함 다운로드`처럼 위험도를 드러냅니다.

### 권한별 화면

owner:

- 원장 관점의 전체 운영 현황, 관리, 리포트, 이력 접근은 적절합니다.
- 다만 `미처리`, `연락 필요`, `체크 필요` 상태 정의가 더 명확해야 매일 열어볼 이유가 강해집니다.

teacher:

- 관리 탭 숨김, 담당 반 필터, 전화번호 비노출은 적절합니다.
- 담당 일정이 없는 날짜의 홈/출석 빈 상태가 더 친절해야 합니다.

assistant:

- 관리 탭 숨김, 담당 반 필터, 전화번호 비노출은 적절합니다.
- 발송 제한이 실제 쓰기 흐름에서 어떻게 보이는지 별도 UAT가 필요합니다. 이 항목은 `T-620`으로 이미 분리되었습니다.

## 3. 버튼/CTA 우선순위 점검

### Primary

Primary는 사용자가 그 화면에서 가장 먼저 해야 하는 행동입니다. 모바일에서는 한 화면에 하나만 강하게 보이는 것이 좋습니다.

권장 Primary:

- 홈: `미처리 보기`, `출석 체크`
- 출석: `도착`, `지각`
- 출석 문자 패널: `기록 저장`, 저장 후 `테스트 발송`
- 문자: 학생 선택 후 `초안 확인` 또는 `문자 작성`
- 전체문자: `발송 전 확인`, 확인 후 `테스트 발송`
- 관리: 각 섹션별 `학생 등록`, `반 만들기`, `수업 시간 등록`

현재 문제:

- 문자 화면에서 `문자 작성`, 사유 버튼, 보강 달력, 최근 기록이 한 화면에 많아 Primary가 흐려질 수 있습니다.
- 전체문자에서 `전체문자 dry-run 발송`이 바로 Primary처럼 보이지만, 발송 전 확인 정보가 약합니다.

### Secondary

Secondary는 보조 이동 또는 필터입니다.

권장 Secondary:

- 홈: `학생 목록 보기`, 날짜 이동
- 출석: `미체크만`, `연락 필요만`, `예외`
- 문자: `반 선택`, `수신자 선택`, `중복 제외`
- 관리: `CSV 등록`, `리포트 다운로드`, `최근 이력 보기`

현재 문제:

- 홈 `목록`은 Secondary인데 의미가 약합니다.
- 출석 `예외`는 Secondary지만 자주 쓰는 버튼이므로 `도착/지각`과 같은 액션 그룹 안에서 현재처럼 가까이 두는 것이 맞습니다.

### Destructive

Destructive는 되돌리기 어렵거나 운영 데이터에 큰 영향을 주는 행동입니다.

권장 Destructive:

- 학생/구성원/스케줄 비활성화
- 스케줄 삭제
- 공유 해제
- 개인정보 원문 포함 CSV 다운로드

현재 문제:

- `삭제`, `비활성`, `공유 해제`, `원문 포함`이 같은 위험도로 보이지 않을 수 있습니다.
- 개인정보 원문 포함 CSV는 단순 옵션보다 warning 성격을 가져야 합니다.

### Passive

Passive는 클릭하면 안 되는 정보입니다.

권장 Passive:

- 홈 요약 숫자 중 클릭 불가 영역
- 리포트 요약 카드
- 최근 변경 이력 row
- 공유된 타 학원 일정 row

현재 문제:

- 카드/row 스타일이 많은 화면에서 클릭 가능한 row와 정보 row의 구분이 아직 약한 곳이 있습니다.
- 공유된 타 학원 일정은 클릭 불가라면 hover/chevron/버튼 느낌을 완전히 없애는 것이 좋습니다.

## 4. 문구 통일안

아래 표현을 기준으로 통일하는 것을 권장합니다.

| 기준 표현 | 의미 | 사용 위치 | 피해야 할 표현 |
| --- | --- | --- | --- |
| 연락 필요 | 결석/지각 등으로 보호자 연락이 필요한 상태 | 홈, 출석 필터, 원장 요약 | 팔로업 필요, followup |
| 체크 필요 | 아직 출석 처리가 안 된 상태 | 출석 현황, 학생 row | pending, 미체크 혼용 |
| 확인 필요 | 결석 확정 전 보류/확인 대상 | 출석 예외 상태 | 체크 필요와 혼용 |
| 미처리 | 원장이 오늘 처리해야 하는 전체 업무 묶음 | 홈 요약 | 확인할 학생 |
| 발송 완료 | 실제 또는 테스트 발송 플로우가 완료된 상태 | 문자 상태 | sent |
| 기록 저장 | 연락 내용을 학생 히스토리에 저장하는 행동 | 문자/출석 문자 패널 | 팔로업 기록 저장 |
| 테스트 발송 | 실제 문자를 보내지 않는 dry-run 발송 | 문자 버튼/상태 | dry-run 발송, dry-run 기록 |
| 보강 후보 | 보강으로 잡을 수 있는 시간 | 보강 달력 | 후보 시간, 선택 가능 시간 혼용 |
| 보강 등록 | 보강 일정을 학생 스케줄에 등록 | 문자 후 보강 등록 | 보강 스케줄 저장 |
| 권한 없음 | 해당 계정이 할 수 없는 행동 | API/UI 에러 | 접근 불가, forbidden |
| 빈 상태 | 해당 날짜/조건에 데이터가 없는 상태 | 홈/출석/관리 | 데이터 없음만 단독 표시 |

추가 권장:

- `학원명 비공개`는 공유 일정 안내에 유지합니다.
- `테스트 모드`는 관리 정책에서 `dry-run` 대신 사용합니다.
- `연락 기록`은 사용자 화면에서 `팔로업` 대신 사용합니다.

## 5. 즉시 수정하면 좋은 항목

### 1. dry-run/팔로업 표현 제거

- 화면: 문자, 출석 문자 패널, 리포트
- 문제: `dry-run`, `팔로업`은 개발/운영 내부 용어라 원장/선생님에게 자연스럽지 않습니다.
- 추천 수정: `dry-run`은 `테스트 발송`, `팔로업`은 `연락 기록`으로 교체합니다.
