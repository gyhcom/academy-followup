# 200명 운영형 파일럿 데이터 계획 - 2026-05-31

## 목적

실제 개인정보 없이 200명 규모의 학원 운영 화면을 검증합니다. 목표는 모바일 390px 기준으로 홈, 출석, 문자, 관리, 공유 일정 화면이 실제 학원 규모에서도 버티는지 확인하는 것입니다.

## Seed 파일

- 파일: `supabase/seed-pilot-200-students.sql`
- 실행 위치: Supabase SQL Editor
- 실행 대상: 파일럿/시연용 DB
- 주의: 실제 학생 개인정보가 들어간 DB에서는 실행하지 않습니다.

## 데이터 구성

- 학원: 더배움프라임영수학원
- 학생: 200명
- 반: 20개
- 학년: 중1, 중2, 중3, 고1, 고2, 무학년제
- 과목: 수학, 영어, 논술, 클리닉
- 수업 요일: 월요일부터 토요일 중심, 일요일 클리닉 일부 포함
- 연락처: `01077xxxxxx`, `01078xxxxxx` 테스트 번호 패턴
- 타 학원 공유 동의: 일부 학생 ON
- 수동 타 학원 수업: 논술, 영어, 수학, 과학 일정 포함
- SaaS 연결 학원 공유: 다솜수학학원, 리딩논술학원, 탑클래스영어학원 더미 연결 포함
- 출석 데이터: 최근/오늘/향후 1주 범위에 출석, 지각, 결석, 확인 필요, 보강, 미체크 상태 혼합
- 문자 기록: 오늘 결석/지각 학생 기준 dry-run 팔로업과 message log 생성

## 화면 확인 기준

### 홈

- 오늘 학원 일정이 10개 이상인 날짜에서도 화면이 너무 길어지지 않는지 확인합니다.
- `공유된 타 학원 일정`이 2개 미리보기와 전체 보기로 자연스럽게 보이는지 확인합니다.
- 공유 일정은 상대 학원명/전화번호 없이 `공유됨`, `학원명 비공개`, `보강 제외`로 표시되어야 합니다.

### 출석

- 학생 8~15명 규모의 수업에서 도착, 지각, 예외 버튼을 빠르게 누를 수 있어야 합니다.
- 미체크/연락 필요 필터가 학생 수가 많아도 동작해야 합니다.

### 문자

- 개별 연락에서 반 선택 후 학생 목록이 과하게 무겁지 않아야 합니다.
- 전체문자에서 대상 수, 실제 발송 수, 중복 제외 수가 이해되어야 합니다.
- byte/SMS/LMS 표시와 2000byte 초과 차단이 유지되어야 합니다.

### 관리

- 학생 200명 기준으로 학생 목록, 검색, 반/스케줄 수정 접근이 가능해야 합니다.
- 원장 시작 순서, 명단, 수업, 직원, 문자, 정책 메뉴 구조가 깨지지 않아야 합니다.

## 제한

- 이 seed는 실제 교직원 Supabase Auth 계정을 새로 만들지 않습니다.
- 선생님/보조 선생님 권한 UAT는 기존 `teacher@test.com`, `assistant@test.com` 계정이 Supabase Auth와 profiles에 준비된 상태에서 확인합니다.
- 실제 SMS 발송은 하지 않고 `sms_dry_run=true`를 유지합니다.
