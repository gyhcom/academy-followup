# 출석부 중심 제품 방향 재정의

## 1. 미팅 배경

월요일 학원 원장 미팅에서 Academy Follow-up의 실제 운영 요구사항을 다시 확인했습니다.

기존 제품 방향은 “수업 직후 후속 연락 관리 도구”에 가까웠습니다. 즉, 결석/지각/숙제 미완료 같은 상황이 생겼을 때 선생님이 빠르게 문자 초안을 만들고 기록하는 데 초점을 두었습니다.

이번 미팅에서는 제품 중심이 더 넓어질 가능성이 확인되었습니다. 원장은 단순 문자 도구보다, 매일 열어보는 출석부 안에서 학생 상태를 확인하고, 결석/지각 문자, 교재비/수강료 안내까지 이어지는 흐름을 원했습니다.

따라서 Academy Follow-up은 홈 대시보드 중심 SaaS보다 “출석부 중심 학원 운영장부”에 가까운 방향을 검토해야 합니다. 다만 전체 학원관리 프로그램과 정면승부하면 수납, 청구, 학부모 앱, 성적, 상담, 리포트까지 범위가 급격히 커지므로, 초기 범위는 출석부와 후속 안내 처리에 제한합니다.

## 2. 핵심 결론

- 제품 중심은 홈 대시보드보다 출석부에 가까워졌습니다.
- 문자 기능은 독립 메뉴라기보다 출석부 안에서 바로 실행되는 흐름이 중요합니다.
- 교재비/수강료 관리는 당장 결제/수납 시스템이 아니라, 금액 기록과 월말 안내 문자 수준으로 제한해야 합니다.
- 전체 학원관리 프로그램과 정면승부하지 않고, 출석부와 후속 연락을 빠르게 처리하는 운영 보조 시스템으로 포지셔닝합니다.

## 3. 요구사항 원문 정리

1. 메뉴 순서에서 출석부를 두 번째로 배치하고 싶다.
2. 출석부 안에 ON/OFF 버튼을 없애고, 출석 / 지각 / 결석 중심으로 단순화하고 싶다.
3. 지각이나 결석인 경우 여러 명을 선택해서 한 번에 일괄 문자 전송하고 싶다.
4. 문자 발송 시 이름을 변수값으로 넣어서 동일한 내용이라도 학생별 이름이 자동 치환되게 하고 싶다.
5. 이름, 수강료, 교재비, 기타 금액 등을 변수로 관리자가 세팅할 수 있게 하고 싶다.
6. 동명이인은 성별 및 휴대폰 번호로 구분하고, 남학생/여학생 아이콘 등으로 구분되면 좋겠다.
7. 날짜도 출석부 화면 상단 또는 화면 안에 명확히 포함되어야 한다.
8. 달력 형식의 출석부가 필요하다.
9. 출석하지 않은 학생을 출석부에서 클릭해서 상태 변경할 수 있어야 한다.
10. 출석부 기준은 학생 row 중심이고, 오른쪽으로 날짜 및 수업시간 정보가 보이는 방식이 필요하다.
11. 교재비가 필요하다.
12. 학생별 교재 선택에 따라 월말에 교재비 합산 금액과 수강료/교재비 결제 필요 안내 문자를 보낼 수 있으면 좋겠다.
13. 학생 개인별 관리 메모가 필요하다.

## 4. 제품 포지션 변경

기존 포지션:

- 수업 직후 후속 연락 관리 도구

변경 후보 포지션:

- 출석부 중심 학원 운영장부
- 결석/지각/교재비 안내까지 출석부에서 처리하는 운영 보조 시스템
- 기존 학원관리 프로그램을 완전히 대체하지 않고, 출석부와 후속 안내를 빠르게 처리하는 보조 도구

## 5. P0 / P1 / P2 우선순위

### P0

- 여러 명 선택 후 일괄 문자

- 성별 아이콘/휴대폰 일부 표시

- 관리자 변수 세팅 화면

## 6. 화면 구조 제안

추천 메뉴 구조:

- 홈
- 출석부
- 문자
- 교재/비용
- 학생
- 관리

초기에는 메뉴를 한 번에 늘리지 않습니다. 실제 구현 순서는 `홈 / 출석부 / 문자 / 관리`를 유지하되, 출석부를 핵심 업무 화면으로 강화하고 교재/비용은 설계 후 별도 메뉴 또는 관리 하위 섹션으로 추가합니다.

출석부 화면 구조:

- 상단: 날짜 / 반 / 수업시간 / 선생님
- 본문: 학생 목록
- 학생 row: 이름 / 성별 / 휴대폰 일부 / 출석상태 / 메모 / 문자 선택
- 상단 또는 하단 액션: 선택 학생 문자 보내기 / 선택 학생 결석 처리 / 선택 학생 지각 처리

PC 구조 후보:

- 좌측: 반/수업 목록
- 중앙: 학생 row 중심 출석부
- 우측: 선택 날짜, 수업시간, 선택 학생 문자/메모 패널

모바일 구조 후보:

- 상단: 날짜와 수업 선택
- 본문: 학생 row 목록
- 하단 sticky action: 선택 학생 상태 변경 / 문자 처리

## 7. 문자 변수 설계 초안

문자 템플릿 변수 후보:

- `{학생명}`
- `{학부모명}`
- `{수업명}`
- `{수업일}`
- `{수업시간}`
- `{출석상태}`
- `{수강료}`
- `{교재비}`
- `{기타비}`
- `{합계금액}`

현재 시스템에는 `{{studentName}}`, `{{academyName}}`, `{{teacherName}}`, `{{className}}` 같은 템플릿 변수가 있습니다. 원장 화면에서는 한국어 변수명인 `{학생명}` 형태가 더 이해하기 쉽습니다. 내부 저장 방식은 기존 변수와 호환되게 유지하되, 화면에서는 한국어 변수 삽입 버튼을 제공하는 방향이 적절합니다.

예시 1:

```text
{학생명} 학부모님, 오늘 {수업명} 수업에 {출석상태} 처리되었습니다.
```

예시 2:

```text
{학생명} 학부모님, 이번 달 수강료는 {수강료}원, 교재비는 {교재비}원입니다. 합계 {합계금액}원 안내드립니다.
```

## 8. 교재비/수강료 범위 제한

초기에는 하지 않을 것:

- 결제 처리
- 카드 결제
- 청구서 발행
- 미납 관리 자동화
- 세금계산서/영수증 처리
- 전체 수납 관리 시스템

초기에는 할 수 있는 것:

- 학생별 수강료 금액 기록
- 학생별 교재 선택
- 학생별 교재비 합산
- 월말 결제 안내 문자 생성
- 안내 문자 발송 기록 저장

이 범위를 넘기면 제품은 “출석부 중심 운영 보조 시스템”이 아니라 “학원 수납 관리 시스템”으로 커집니다. 파일럿 단계에서는 금액 계산과 안내 문자까지만 다룹니다.

## 9. 기존 기능과의 연결

- 출석 체크: 출석부 중심 기능으로 강화합니다.
- 결석/지각 문자: 다중 선택 일괄 문자로 확장합니다.
- 문자 템플릿: 변수 기반 템플릿으로 확장합니다.
- 학생/반/선생님 관리: 출석부 기준 데이터로 유지합니다.
- 리포트: 월별 출석/문자/교재비 리포트로 확장할 수 있습니다.
- 권한: 원장/선생님 출석부 접근 제어에 계속 필요합니다.
- 감사 로그: 학생/수업/문자/금액 변경 이력 확인에 확장할 수 있습니다.

## 10. 리스크

- 수강료/교재비 기능이 수납 관리 시스템으로 커질 위험이 있습니다.
- 달력형 출석부 구현 범위가 커질 위험이 있습니다.
- 문자 변수 기능이 템플릿 엔진 수준으로 커질 위험이 있습니다.
- 동명이인/성별/휴대폰 표시가 개인정보 노출 위험으로 이어질 수 있습니다.
- 출석부와 문자 기능이 너무 강하게 결합되어 화면과 API가 복잡해질 수 있습니다.
- 기존 Spring Boot 전환 문서의 T-636~T-641 번호와 이번 미팅 후속 티켓 번호가 충돌합니다. GitHub Issue는 미팅 요청 번호를 유지하되, 문서에서는 “출석부 방향 후속 이슈”로 별도 구분합니다.
