# 파일럿 운영 가이드

이 문서는 더배움프라임영수학원 데모/파일럿을 실제 운영처럼 검증하기 위한 기준입니다.

목표는 기능을 많이 보여주는 것이 아니라, 선생님과 원장이 매일 반복하는 출석, 결석, 지각, 보강, 문자 안내 흐름이 기존 방식보다 빠르고 안전한지 확인하는 것입니다.

## 1. 파일럿 목적

Academy Follow-up의 파일럿 목적은 아래 한 문장으로 정의합니다.

> 수업 직후 발생하는 확인 필요, 지각, 결석, 보강 안내 업무를 한 화면에서 처리하고, 원장이 처리 누락을 확인할 수 있게 한다.

파일럿에서 검증할 핵심 가설은 다음과 같습니다.

- 선생님은 수업 직후 10초 안에 학생 상태를 기록할 수 있다.
- 결석/지각 학생은 출석부에서 바로 문자 초안으로 연결된다.
- 보강이 필요한 학생은 학생 스케줄을 보고 날짜와 시간을 바로 선택할 수 있다.
- 원장은 오늘 미처리 학생과 연락 상태를 따로 카톡방을 뒤지지 않고 확인할 수 있다.
- dry-run 발송만으로도 실제 운영 흐름을 충분히 검증할 수 있다.

## 2. 파일럿 범위

### 포함

- 원장 계정 로그인
- 선생님 계정 로그인
- 반별 출석부
- 학생별 상태 체크
- 상태: 출석, 확인 필요, 지각, 결석, 보강
- 결석/지각 문자 초안 생성
- 학부모/학생 수신자 선택
- dry-run 발송 기록
- 학생별 팔로업 히스토리
- 날짜별 보강 시간 선택
- 보강 문자 생성
- 보강 스케줄 자동 등록
- 원장용 오늘 현황 확인

### 제외

- 실제 전체 학원 문자 발송 전환
- 학부모 앱
- 학생 앱
- 수납/청구
- 온라인 강의
- 문제은행
- 전체 성적 분석
- 카카오 알림톡
- 복잡한 드래그 앤 드롭 캘린더

## 3. 파일럿 대상

초기 파일럿은 전체 학원을 한 번에 적용하지 않고 작은 범위로 시작합니다.

추천 범위:

- 기간: 2주
- 참여자: 원장 1명, 선생님 2~3명
- 대상 반: 2~3개
- 대상 학생: 30명 내외
- 사용 기기: 선생님 모바일, 원장 PC 또는 태블릿

처음부터 원생 200명 전체를 넣지 않습니다. 먼저 실제 업무 흐름이 맞는지 확인한 뒤 CSV 일괄 등록과 전체 확장을 진행합니다.

## 4. 하루 운영 흐름

### 4.1 수업 시작 전

1. 선생님이 로그인합니다.
2. 오늘 수업 반을 선택합니다.
3. 학생 목록과 예정 수업 시간을 확인합니다.
4. 누락된 학생이나 반 정보가 있으면 원장/관리자에게 공유합니다.

### 4.2 수업 시작 후 5~10분

1. 아직 도착하지 않은 학생을 `확인 필요`로 체크합니다.
2. 이 단계에서는 바로 결석 문자를 보내지 않습니다.
3. 늦게 도착한 학생은 `지각`으로 변경합니다.
4. 실제 결석으로 확정된 학생만 `결석`으로 변경합니다.

### 4.3 결석/지각 확정 후

1. 결석 또는 지각 학생을 선택합니다.
2. 문자 초안을 확인합니다.
3. 수신자를 학부모, 학생, 학부모+학생 중 선택합니다.
4. 문구를 학원 말투에 맞게 수정합니다.
5. dry-run 발송 또는 팔로업 기록 저장을 진행합니다.
6. 학생 히스토리에 기록이 남는지 확인합니다.

### 4.4 보강이 필요한 경우

1. 학생의 주간 스케줄과 날짜별 일정을 확인합니다.
2. 보강 가능한 날짜를 선택합니다.
3. 시작 시간과 종료 시간을 직접 입력합니다.
4. 보강 안내 문자 초안을 생성합니다.
5. dry-run 발송을 완료합니다.
6. `스케줄에 입력`을 눌러 보강 일정을 자동 등록합니다.
7. 학생 상세에서 보강 일정이 보이는지 확인합니다.

### 4.5 원장 확인

1. 원장은 오늘 출석 현황을 확인합니다.
2. `확인 필요`, `지각`, `결석` 학생을 확인합니다.
3. 아직 문자 초안이나 발송 기록이 없는 학생을 확인합니다.
4. 선생님별 처리 상태를 확인합니다.
5. 반복적으로 누락되는 반이나 선생님이 있는지 확인합니다.

## 5. 핵심 시나리오

### 시나리오 A. 수업 시작 10분 후 미도착 학생

```text
16:00 수업 시작
16:10 학생 미도착
선생님이 확인 필요로 체크
16:15 학생 도착
지각으로 변경
지각 안내 문자 초안 생성
dry-run 발송
학생 히스토리에 기록 저장
```

검증 포인트:

- `확인 필요` 상태가 결석과 명확히 구분되는가?
- 늦게 도착한 학생을 쉽게 지각으로 바꿀 수 있는가?
- 지각 문자가 학부모에게 부담스럽지 않은 표현인가?

### 시나리오 B. 결석 확정 후 보강 일정 잡기

```text
학생 결석 확정
결석 문자 초안 생성
학생 스케줄 확인
보강 날짜와 시간 선택
보강 안내 문자 생성
dry-run 발송
보강 스케줄 자동 등록
```

검증 포인트:

- 기존 카톡/엑셀보다 보강 후보 시간을 찾기 쉬운가?
- 보강 문자를 보낸 뒤 스케줄 등록을 잊지 않게 되는가?
- 같은 시간대 보강이 중복 등록되지 않는가?

### 시나리오 C. 원장 미처리 확인

```text
원장이 오늘 현황 확인
확인 필요/지각/결석 학생 확인
연락 미처리 학생 확인
담당 선생님 확인
필요 시 후속 조치
```

검증 포인트:

- 원장이 카톡방을 보지 않아도 오늘 누락을 파악할 수 있는가?
- 어떤 선생님이 어떤 학생을 처리했는지 알 수 있는가?
- 유료 기능으로 볼 만큼 가치가 있는가?

## 6. 운영 규칙

### 문자 발송 규칙

- 파일럿 초반에는 `SMS_DRY_RUN=true`를 기본으로 유지합니다.
- 실제 학부모 번호로 발송하기 전에는 테스트 번호로만 검증합니다.
- `확인 필요` 상태에서는 결석 문자를 바로 보내지 않습니다.
- 결석/지각 확정 후에도 미리보기와 수신자 확인 단계를 거칩니다.
- 학부모+학생 동시 발송 시 수신자 목록을 다시 확인합니다.

### 데이터 입력 규칙

- 초기에는 30명 내외 학생만 입력합니다.
- 학생명, 반, 학년, 학부모 전화번호를 우선 입력합니다.
- 학생 전화번호는 선택값으로 둡니다.
- 외부 일정은 보강 판단에 필요한 수준만 입력합니다.
- 삭제보다 비활성화를 우선합니다.

### 피드백 수집 규칙

- 선생님에게 긴 설문을 요구하지 않습니다.
- 실제 수업 후 사용 직후 1~2개 질문만 확인합니다.
- 원장에게는 하루가 끝난 뒤 미처리 확인 화면이 도움이 됐는지 확인합니다.

## 7. 파일럿 회의 질문

### 원장 질문

- 오늘 결석/지각/확인 필요 학생을 바로 파악할 수 있었나요?
- 기존 카톡방 공유 방식보다 누락 확인이 쉬웠나요?
- 원장 입장에서 돈을 낼 만한 화면은 무엇인가요?
- 오늘 현황, 학생별 히스토리, 선생님별 처리 현황 중 무엇이 가장 중요한가요?
- 실제 운영에 쓰려면 반드시 추가되어야 할 기능은 무엇인가요?

### 선생님 질문

- 수업 직후 학생 상태를 체크하는 흐름이 불편하지 않았나요?
- 결석/지각 문자 초안이 실제로 쓸 만했나요?
- 보강 날짜와 시간을 고르는 방식이 기존 방식보다 편했나요?
- 모바일에서 조작하기 어려운 부분이 있었나요?
- 너무 많은 클릭이 필요한 부분은 어디였나요?

## 8. 파일럿 종료 후 판단

2주 후 아래 세 가지를 기준으로 다음 단계를 결정합니다.

### 계속 개발

- 선생님이 주 3회 이상 실제 사용
- 원장이 오늘 현황을 반복 확인
- 결석/지각/보강 흐름이 기존 방식보다 빠르다는 피드백 확보
- 오발송이나 중복 발송 사고 없음

### 범위 축소

- 문자 초안은 쓰지만 보강 스케줄은 거의 쓰지 않음
- 원장 현황 화면을 보지 않음
- 모바일 사용성이 낮아 실제 수업 후 사용이 어려움

### 중단 또는 방향 전환

- 선생님이 기존 카톡/엑셀보다 불편하다고 판단
- 원장이 운영 통제 가치를 느끼지 못함
- 실제 업무가 제품 가설과 크게 다름

## 9. 다음 단계

파일럿에서 긍정 신호가 확인되면 다음 순서로 확장합니다.

1. CSV 학생 일괄 등록
2. 선생님 계정/권한 관리
3. 원장 미처리 보드 고도화
4. 실제 SOLAPI 발송 제한 테스트
5. 무료/유료 기능 경계 확정
6. 두 번째 학원 파일럿 제안
