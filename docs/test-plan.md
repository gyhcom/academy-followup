# 기능 테스트 계획

이 문서는 친구 원장에게 보여주기 전 Academy Follow-up의 핵심 흐름을 점검하기 위한 체크리스트입니다.

## 1. 현재 연결 상태

확인된 항목:

- Vercel Production 배포 완료
- Supabase Production 환경변수 연결
- `/login` 로그인 화면 표시
- 테스트 계정 로그인 가능
- `/app`에서 더배움프라임영수학원 워크스페이스 표시
- `/app`에서 반/학생 선택 및 문자 초안 확인 가능
- `/app`의 출석부 탭에서 날짜/수업별 학생 출석 상태 체크 가능

## 1-1. Supabase Seed 점검

- `students.student_phone` 컬럼 존재 여부 확인 SQL이 동작함
- `supabase/seed.sql`만 실행하면 MVP 최소 학생 수로 유지됨
- 볼륨 테스트 데이터는 `supabase/seed-volume.sql`을 실행해야만 추가됨
- seed 실행 후 반/학생/스케줄/출석 기록 수 확인 SQL이 동작함
- 적용 순서는 `docs/supabase-setup.md`를 기준으로 확인함

## 3. 파일럿 시연 핵심 시나리오

시연 목표는 "출석부에서 미도착 학생 체크 -> 결석/지각 문자 초안 -> dry-run 발송 -> 기록 확인"입니다.

시나리오:

1. 원장 계정으로 로그인
2. 출석부 탭 선택
3. 오늘 날짜와 16:30 수업 선택
4. 미도착 학생을 `확인 필요`로 체크
5. 실제 결석 학생을 `결석`으로 변경
6. 결석 문자 초안 생성
7. 문구 일부 수정
8. dry-run 발송
9. 오늘 팔로업에 기록 표시
10. 학생 히스토리에 기록 표시

## 4. 로그인/세션 테스트

- 비로그인 상태에서 `/app` 접근 시 `/login`으로 이동
- 틀린 비밀번호 입력 시 오류 메시지 표시
- 올바른 계정 로그인 시 `/app` 이동
- 로그아웃 후 다시 `/app` 접근 시 차단
- Production redeploy 후 기존 세션이 과도하게 깨지지 않는지 확인

## 5. 데이터 조회 테스트

- 원장 계정은 학원 전체 반을 볼 수 있음
- 선생님 계정은 담당 반 중심으로 볼 수 있음
- 반 선택 시 학생 목록이 해당 반 기준으로 바뀜
- 학생이 없는 반은 빈 상태를 보여줌
- 학부모 전화번호는 설정에 따라 마스킹됨

## 5-1. 구성원 관리 테스트

- 원장/관리자 계정은 관리 탭에서 구성원을 등록할 수 있음
- 구성원 등록 시 Supabase Auth 사용자와 `profiles`가 같은 ID로 연결됨
- 등록된 선생님 계정은 이메일과 임시 비밀번호로 로그인 가능
- 구성원 목록에서 이름, 이메일, 전화번호, 역할, 활성/비활성 상태가 표시됨
- 원장/관리자 계정은 구성원 역할과 상태를 수정할 수 있음
- 비활성 구성원은 `/app`에서 워크스페이스 화면 대신 비활성 안내를 봄
- 선생님 계정은 관리 탭과 `/api/members`에 접근할 수 없음
- 비로그인 `GET/POST/PATCH /api/members`는 401을 반환함

## 5-2. 역할/권한 테스트

- owner/manager는 운영 보드, 출석부, 관리 화면에서 학원 전체 데이터를 볼 수 있음
- teacher/assistant는 담당 반 데이터만 볼 수 있음
- teacher/assistant는 관리 탭이 보이지 않음
- teacher/assistant가 학생/반 관리 API를 호출하면 403을 반환함
- teacher는 담당 반 팔로업 문자를 발송할 수 있음
- assistant는 `allow_assistant_send=false`일 때 실제 발송 API에서 403을 반환함
- assistant는 `allow_assistant_send=true`일 때 담당 반 팔로업만 발송할 수 있음

## 5-3. 학원 설정 테스트

- 원장/관리자는 관리 탭에서 학원명, 발신명, 발신번호를 수정할 수 있음
- dry-run 토글을 저장하면 `/api/messages/send`의 실제 발송 차단 여부에 반영됨
- 중복 발송 방지 시간을 저장할 수 있음
- 보조 선생님 발송 허용 여부를 저장할 수 있음
- 선생님 계정은 설정 화면에 접근할 수 없음
- 비로그인 `PATCH /api/academy-settings`는 401을 반환함

## 6. 문자 미리보기 테스트

- 학생과 사유를 선택해야 미리보기 생성 가능
- 학원명, 학생명, 선생님명이 템플릿에 치환됨
- 템플릿이 없으면 명확한 오류 표시
- 본문 수정 후 발송 요청에는 수정본이 사용됨
- 수신자를 학부모, 학생, 학부모+학생 중 선택할 수 있음
- 학생 연락처가 없는 학생은 학생 발송 옵션이 비활성화됨
- dry-run 발송 로그에는 실제 선택된 수신자 유형이 저장됨

## 7. 출석부 테스트

- 날짜를 바꾸면 해당 날짜 출석 기록을 다시 불러옴
- 수업 시간 목록이 시간순으로 정렬됨
- 같은 수업의 학생이 이름순으로 표시됨
- 출석/지각/결석/확인 필요/보강 상태를 저장할 수 있음
- 저장 후 새로고침해도 선택한 상태가 유지됨
- `확인 필요`는 결석 확정 전 대기 상태로 남길 수 있음
- `결석` 선택 시 결석 문자 초안이 표시됨
- `지각` 선택 시 지각 문자 초안이 표시됨
- 문자 본문을 수정한 뒤 팔로업 기록으로 저장 가능
- 저장된 팔로업이 출석 기록과 연결됨
- 오늘 출석 현황에서 상태별 카운트가 표시됨
- 수업 시간별 반 요약에서 미체크 수를 확인 가능
- 결석/지각/확인 필요 학생이 연락 대상 목록에 표시됨
- 팔로업 저장/발송 상태가 연락 처리 상태에 반영됨
- 선생님 계정은 담당 반 출석부만 수정 가능
- 원장 계정은 전체 반 출석부를 확인 가능

## 7-1. 원장 오늘 미처리 보드 테스트

- owner/manager는 운영 보드 상단에서 오늘 미처리 보드를 볼 수 있음
- teacher/assistant는 원장 미처리 보드가 보이지 않음
- 결석/지각/확인 필요/보강/미체크 학생이 목록에 표시됨
- 발송 완료된 학생은 `발송 완료`, 아직 처리되지 않은 학생은 `미처리`로 표시됨
- 반 필터와 상태 필터가 모바일에서도 가로 overflow 없이 동작함
- 처리 완료 항목은 미처리 항목보다 아래에 정렬됨

## 8. 날짜별 보강 테스트

- 운영 보드에서 학생을 선택하면 월간 보강 달력이 표시됨
- 이전/다음 달 버튼으로 월을 이동할 수 있음
- 날짜 셀을 클릭하면 해당 요일의 정규 수업/외부 일정이 표시됨
- 시작/종료 시간이 올바르지 않으면 보강 후보를 선택할 수 없음
- 날짜와 시간을 선택하면 보강 문자 초안에 해당 날짜/시간이 반영됨
- 보강 팔로업 저장 후 dry-run 발송을 완료해야 `스케줄에 입력`이 가능함
- `스케줄에 입력`을 누르면 날짜 지정 `makeup` 일정이 생성됨
- 같은 학생/같은 날짜/같은 시간 보강은 중복 등록되지 않음
- 자동 등록된 보강 일정은 관리 탭 학생 상세에서 확인하고 수정할 수 있음

## 8-1. 반 스케줄 일괄 등록 테스트

- 관리 탭 반 목록에서 `스케줄` 버튼을 누르면 일괄 등록 폼이 표시됨
- 반, 요일, 시작/종료 시간, 제목을 기준으로 재원 학생 전체에게 반복 스케줄이 생성됨
- 같은 요일/시간의 활성 스케줄이 이미 있는 학생은 건너뜀
- 저장 후 등록 수와 건너뛴 수가 안내됨
- 일괄 등록된 스케줄은 학생 관리 상세와 운영 보드 주간표에 표시됨
- 비로그인 `POST /api/student-schedules/bulk`는 401을 반환함

## 9. 발송 안전 테스트

- `sms_dry_run=true`인 학원에서는 실제 SOLAPI 호출 금지
- dry-run도 `followups`와 `message_logs`에 기록
- 학부모+학생 발송은 수신자별로 `message_logs`가 분리 저장됨
- `/api/messages/send`는 같은 학생/같은 사유/같은 수신자 유형의 당일 또는 `duplicate_guard_minutes` 이내 중복 발송을 409로 차단
- 중복 차단 시 dry-run/실발송 모두 SOLAPI 호출과 `message_logs` 추가 생성이 없어야 함
- 다른 학생, 다른 사유, 다른 수신자 유형은 중복 차단 없이 발송 가능
- 발송 실패 시 `failed` 상태와 오류 메시지 저장
- 서버 로그에 secret key나 학부모 전화번호 원문을 과도하게 남기지 않음

## 10. 멀티테넌트 테스트

여러 학원으로 확장하기 전 반드시 확인합니다.

- 테스트 학원 A/B 생성
- 각 학원에 별도 사용자/profile/classes/students 생성
- 학원 A 계정으로 학원 B 학생 데이터 조회 불가
- preview/send API에서도 요청 대상 데이터의 `academy_id` 검증

## 11. 모바일 사용성 테스트

휴대폰 접속을 1차 사용 환경으로 보고 확인합니다.

- 390px 폭에서 페이지 전체 가로 스크롤 없음
- 반 선택 칩을 좌우로 넘겨도 화면이 흔들리지 않음
- 학생 목록이 너무 길어도 문자 작업 영역까지 바로 접근 가능
- 학생 카드의 이름, 학년, 연락처 마스킹이 겹치지 않음
- 사유 버튼과 dry-run 버튼 높이가 엄지 터치에 충분함
- iPhone 하단 홈 인디케이터와 마지막 버튼이 붙지 않음
- 출석부 상태 버튼이 390px 폭에서 겹치지 않고 좌우 스크롤로 조작 가능
- 수업 시간 목록과 학생 체크 영역이 모바일에서 과도하게 좁아지지 않음
- 보강 달력의 날짜 셀이 390px 폭에서 겹치지 않음
- 보강 달력의 시간 입력이 390px 폭에서 세로로 자연스럽게 표시됨
- 보강 문자 작성 후 하단 패널에서 스케줄 등록 버튼까지 접근 가능함

## 12. 태블릿/PC 반응형 테스트

- iPad 세로/가로 폭에서 상단 학원명, 탭, 로그아웃 버튼이 겹치지 않음
- 운영 보드 중간 폭에서는 학생 목록과 보강 달력이 2열로 보이고 문자 패널은 아래로 내려감
- PC 넓은 폭에서는 학생 목록, 보강 달력, 문자 패널이 3열로 보임
- 출석 현황의 반별 요약 표가 카드 밖으로 밀리지 않음
- 출석 현황 우측 연락 대상 패널은 충분히 넓은 화면에서만 옆으로 붙음
- 관리 요약 카드가 iPad 폭에서 과도하게 세로로 늘어나지 않음
- 긴 반 이름, 학원명, 이메일은 말줄임 또는 자연스러운 줄바꿈으로 처리됨
