# 친구 원장 데모 시나리오

이 문서는 더배움프라임영수학원 원장에게 보여줄 첫 데모 흐름입니다.

목표는 기능을 많이 보여주는 것이 아니라, 아래 한 문장을 3분 안에 납득시키는 것입니다.

> 선생님이 반별 출석을 체크하면, 원장은 연락 누락을 보고, 결석/지각/보강 연락은 기록까지 한 흐름으로 남긴다.

## 1. 데모 전 상태

데모 전에 아래 상태를 맞춥니다.

- `owner@test.com` 원장 계정으로 로그인 가능
- `uat-teacher@test.com` 선생님 계정으로 로그인 가능
- `uat-assistant@test.com` 보조 선생님 계정으로 로그인 가능
- `sms_dry_run=true`
- UAT 반 3개 이상
- UAT 학생 30명 내외
- 오늘 날짜 기준 정규 수업 스케줄 있음
- 보강 달력에 정규 수업/외부 일정이 보임

데모 중 실제 SMS 발송은 하지 않습니다. `dry-run` 기록까지만 보여줍니다.

## 2. 3분 데모 흐름

### 0:00 - 0:20 문제 확인

말할 내용:

```text
지금은 선생님이 결석/지각을 카톡방에 공유하고,
데스크나 원장이 다시 확인해서 연락하는 흐름이 있잖아.
이 화면은 그 흐름을 출석부에서 바로 처리하게 하는 용도야.
```

보여줄 화면:

- `/login`
- 원장 계정 로그인

확인 포인트:

- 학원명 `더배움프라임영수학원` 표시
- 원장 계정으로 들어왔다는 느낌

### 0:20 - 0:50 원장 미처리 보드

보여줄 화면:

- 운영 탭 상단 `원장 오늘 미처리 보드`

말할 내용:

```text
원장은 오늘 연락이 필요한 학생만 먼저 보면 돼.
결석, 지각, 확인 필요, 보강 예정인데 아직 처리 안 된 학생이 여기 모여.
```

보여줄 것:

- 연락 누락 가능 학생 수
- 미처리/완료 수
- 상태 필터
- 반 필터

강조:

- 카톡방을 계속 뒤지는 대신 “오늘 처리할 학생 목록”이 생김

### 0:50 - 1:30 반별 출석부

보여줄 화면:

- 출석 탭
- 오늘 날짜
- `UAT 중2 수학 A반`

말할 내용:

```text
선생님은 수업 시간별로 학생 도착 여부만 누르면 돼.
바로 결석 확정이 애매하면 확인 필요로 잠깐 둘 수 있어.
```

시연 행동:

1. 수업 시간 선택
2. 학생 한 명 `확인 필요`
3. 학생 한 명 `지각`
4. 학생 한 명 `결석`

확인 포인트:

- 오늘 출석 요약 카운트 변경
- 오늘 연락 필요 목록에 결석/지각/확인 필요가 올라옴
- 잘못 눌러도 다른 상태로 다시 바꿀 수 있음

### 1:30 - 2:05 결석/지각 문자

보여줄 화면:

- 결석 또는 지각 학생 선택 후 문자 패널

말할 내용:

```text
결석이나 지각으로 확정하면 학부모 문자 초안이 바로 떠.
그대로 보내는 게 아니라 선생님이 문구를 한번 보고 수정할 수 있어.
```

시연 행동:

1. 결석 학생 선택
2. 문자 초안 확인
3. 수신자 `학부모`, `학생`, `학부모+학생` 버튼 보여주기
4. 본문 한 줄 수정
5. 팔로업 기록 저장
6. 문자 발송 테스트 클릭

확인 포인트:

- 실제 문자는 보내지 않음
- `dry-run 기록 완료`
- 학생 히스토리에 기록 추가
- 같은 학생/사유/수신자 중복 발송은 막힘

### 2:05 - 2:40 보강 날짜/시간

보여줄 화면:

- 운영 탭
- 학생 선택
- 보강 달력

말할 내용:

```text
보강은 그냥 월화수목금 같은 반복 시간이 아니라,
실제 날짜를 찍고 그날 학생 수업이나 외부 일정을 보면서 잡는 방식이야.
```

시연 행동:

1. 학생 한 명 선택
2. 보강 달력에서 날짜 확인
3. 기존 정규 수업/외부 일정 확인
4. 보강 가능 시간 입력
5. 보강 문자 후보 선택
6. 문자 저장
7. dry-run 발송
8. 스케줄에 입력

확인 포인트:

- 발송 전에는 보강 스케줄 등록 버튼이 비활성화됨
- 발송 테스트 후 스케줄 등록 가능
- 같은 날짜/시간 중복 등록은 막힘
- 학생 주간 스케줄에 1회성 보강이 보임

### 2:40 - 3:00 기록 확인

보여줄 화면:

- 최근 팔로업
- 원장 미처리 보드

말할 내용:

```text
핵심은 문자 하나 보내는 게 아니라,
누가 어떤 학생에게 어떤 사유로 연락했는지가 남는 거야.
그래서 원장은 오늘 누락된 연락만 보면 돼.
```

확인 포인트:

- 최근 팔로업 최신순
- 발송/저장 상태 구분
- 오늘 미처리 보드에서 처리 상태 확인 가능

## 3. 보조 화면 확인

시간이 남을 때만 보여줍니다.

### 선생님 계정

계정:

```text
uat-teacher@test.com
```

보여줄 것:

- 관리 탭이 보이지 않음
- 배정된 반만 보임
- 출석부와 문자 흐름은 사용 가능

### 보조 선생님 계정

계정:

```text
uat-assistant@test.com
```

보여줄 것:

- 배정 반이 없으면 데이터가 보이지 않음
- “담당 반이 배정되지 않았습니다” 안내가 보임

말할 내용:

```text
권한에 따라 볼 수 있는 반과 메뉴를 제한할 수 있어.
```

## 4. 데모 중 말하지 않을 것

- “전체 200명 바로 도입하자”
- “실제 SMS 발송까지 이미 완성됐다”
- “수납/성적/문제은행까지 다 된다”
- “AI가 알아서 다 처리한다”
- “다른 학원 학생 여부를 전화번호로 알 수 있다”

대신 이렇게 말합니다.

```text
처음에는 몇 개 반만 넣어서 수업 후 연락 흐름이 진짜 줄어드는지 보자.
실제 문자는 처음엔 dry-run으로 확인하고, 괜찮으면 발송 설정을 붙이면 돼.
```

## 5. 데모 후 질문

데모 직후에는 긴 설문을 하지 않습니다. 아래 5개만 묻습니다.

1. 선생님이 수업 직후 실제로 누를 수 있을 만큼 단순한가?
2. 결석/지각 문자를 지금보다 빨리 만들 수 있을 것 같은가?
3. 원장 입장에서 오늘 미처리 보드가 카톡방 확인을 줄여주는가?
4. 보강 날짜/시간 선택 방식이 실제 운영 방식과 맞는가?
5. 파일럿을 한다면 먼저 넣을 반 1~2개는 어디가 좋은가?

## 6. 데모 성공 기준

긍정 신호:

- “이건 우리 데스크가 쓰면 편하겠다”
- “결석 연락 누락은 줄겠다”
- “반 몇 개만 넣어서 테스트해보자”
- “선생님한테 한번 보여줘도 되겠다”

부정 신호:

- “수업 직후 누르기엔 복잡하다”
- “보강은 이 방식으로 안 잡는다”
- “카톡방이 더 빠르다”
- “학생 데이터 넣는 게 너무 부담스럽다”

## 7. 다음 액션

데모가 긍정적이면 바로 전체 도입을 제안하지 않습니다.

추천 다음 단계:

1. 반 1~2개만 파일럿 범위로 정함
2. 학생 30명 내외 실제 데이터 입력 방식 결정
3. 선생님 1~2명 계정 생성
4. 1주일은 dry-run으로 기록만 남김
5. 불편한 문구와 버튼을 고친 뒤 실제 발송 여부 결정
