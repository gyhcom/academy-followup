## 1. 현재 완료된 핵심 기능

### 출석

- 수업별 출석부가 구현되어 있습니다.
- 날짜 기준으로 해당 요일 수업을 선택하고 학생별 상태를 기록할 수 있습니다.
- `도착` 토글은 `present`, OFF는 `pending`으로 처리합니다.
- `지각`, `결석`, `확인 필요`, `보강` 상태를 기록할 수 있습니다.
- 지각/결석은 문자 초안 흐름으로 이어집니다.
- 모바일 390px을 기준으로 출석 UX를 여러 차례 압축했습니다.

남은 판단:
- 실제 200명 데이터에서 한 수업 학생 수가 늘어났을 때 조작 속도와 저장 체감은 아직 운영 검증이 필요합니다.

### 보강

- 학생별 주간/일회성 스케줄을 관리합니다.
- 보강 날짜/시간 선택과 문자 초안 생성 흐름이 있습니다.
- 문자 후 보강 스케줄 자동 등록 흐름이 있습니다.
- 우리 학원 정규 수업, 개인/기타 일정, 수동 타 학원 수업, 연결 학원 공유 일정과 겹치는 시간은 보강 후보에서 차단하도록 설계되어 있습니다.
- 타 학원 스케줄 공유 동의, 자동 연결, 익명 표시, 수동 타 학원 수업 등록까지 구현되어 있습니다.

남은 판단:
- 원장 화면에서 “왜 이 시간이 보강 불가인지”를 실제 데이터로 확인해야 합니다.
- 연결 학원 공유 일정이 많을 때 홈/학생 상세/보강 화면에서 혼란이 없는지 운영 검증이 필요합니다.

### 문자/dry-run

- 학생별 문자 초안 생성, 수정, 저장이 구현되어 있습니다.
- 학부모/학생/둘 다 수신자 선택이 가능합니다.
- `sms_dry_run=true` 기준으로 실제 발송 없이 로그를 남길 수 있습니다.
- 문자 길이 byte, SMS/LMS 예상, 2000byte 초과 차단이 구현되어 있습니다.
- 전체문자와 중복 수신자 제외가 구현되어 있습니다.
- 중복 발송 방지 시간 설정이 있습니다.

남은 판단:
- 실제 SOLAPI 발송은 아직 파일럿 전 제한 테스트가 필요합니다.
- 중복 제외는 실제 가족/형제/중복 연락처 데이터에서 재검증해야 합니다.

### 구성원

- 구성원 등록/수정이 구현되어 있습니다.
- 원장 화면에서는 이메일 입력을 숨기고 직위 중심으로 보이게 정리되어 있습니다.
- `owner`, `manager`, `teacher`, `assistant` 역할 모델이 있습니다.
- 선생님/보조 선생님 테스트 계정 기준 검증 문서가 있습니다.

남은 판단:
- 실제 선생님 2~3명 계정으로 담당 반 기준 필터가 운영 데이터에서도 맞는지 재검증해야 합니다.

### 권한

- 학원 내부 권한은 `owner/manager/teacher/assistant`로 분리되어 있습니다.
- 원장/관리자는 전체 데이터와 관리 탭에 접근합니다.
- 선생님/보조 선생님은 담당 반 중심으로 제한됩니다.
- 보조 선생님 실제 문자 발송은 설정값을 따릅니다.
- 플랫폼 관리자와 학원 워크스페이스가 분리되어 있습니다.

남은 판단:
- API별 권한 검증이 대부분 들어가 있으나, 파일럿 전 `teacher/assistant` 계정으로 직접 누르는 UAT가 필요합니다.
- 담당 반이 없는 계정의 빈 상태 UX 확인이 필요합니다.

### 설정

- 학원명, 발신 정보, dry-run 여부, 중복 방지 시간, 보조 선생님 발송 허용 여부를 관리합니다.
- 실제 발송 전 `sms_dry_run=true` 유지가 문서화되어 있습니다.
- 플랫폼 콘솔에서 학원 생성/상태 관리 흐름이 있습니다.

남은 판단:
- Production 환경변수와 Supabase 운영 DB 값이 실제 배포에서 일치하는지 체크리스트화가 필요합니다.

### 원장 보드

- 로그인 후 홈 화면에 날짜 기준 운영 요약이 있습니다.
- 오늘 일정판이 있고, 학원 일정과 공유된 타 학원 일정을 구분합니다.
- 확인할 학생, 연락 전, 연락 완료 흐름이 있습니다.
- 원장은 출석/문자/관리로 빠르게 이동할 수 있습니다.

남은 판단:
- 원장이 매일 열어볼 “오늘 미처리”의 강도가 아직 약할 수 있습니다.
- 실제 운영 중 미처리 학생이 쌓였을 때 원장이 어떤 순서로 처리해야 하는지 더 명확히 보여줄 필요가 있습니다.

### 학생 히스토리

- 학생별 팔로업 히스토리를 최신순으로 볼 수 있습니다.
- 출석에서 결석/지각을 선택한 뒤 문자 저장/발송 기록으로 이어집니다.
- 최근 연락 기록은 원장/선생님이 확인할 수 있습니다.

남은 판단:
- 원장이 학생 상세에서 “최근 누락/연락/보강”을 한눈에 보는 정도는 실제 데이터로 확인해야 합니다.

### 모바일 UX

- 제품 기준이 PC 대시보드가 아니라 모바일 390px 운영 앱으로 전환되어 있습니다.
- 홈, 출석, 문자, 관리 화면의 모바일 흐름을 여러 번 보정했습니다.
- 하단 탭 중심 내비게이션입니다.
- PC/iPad는 보조 화면으로 보고 가로 overflow와 주요 버튼 접근만 확인합니다.

남은 판단:
- 실제 원장/선생님이 1주일 쓰며 느끼는 클릭 수, 문구, 스크롤 피로도를 확인해야 합니다.

## 2. 6월 중순 MVP 기능 동결 기준으로 부족한 기능

### 기능 부족

1. 실제 SMS 제한 발송 준비
   - dry-run은 충분하지만, 6월 말 시연 전에는 테스트 번호 1개로 실제 수신 확인이 필요합니다.
   - 단, 6월 중순 기능 동결 전에는 실제 발송 구현보다 발송 전환 체크리스트와 보호 장치 확인이 우선입니다.

2. 200명 규모 운영형 데이터 검증
   - 현재 30명 구조 CSV와 여러 demo seed는 있으나, 200명 규모에서 홈/출석/문자/관리의 밀도 검증이 부족합니다.
   - 실제 운영 규모에 가까운 데이터에서 리스트, 필터, 검색, 바텀시트가 버티는지 확인해야 합니다.

3. 선생님/보조 선생님 실계정 UAT
   - 권한 모델은 있지만 실제 담당 반 배정, 전화번호 비노출, 문자 제한을 운영형 데이터로 다시 확인해야 합니다.

4. 운영 증거/CSV 내보내기
   - main 기준으로는 아직 예정입니다.
   - PR #19에 리포트/CSV 내보내기가 포함되어 있으므로 새 작업으로 만들지 말고 PR #19 병합 여부를 먼저 결정해야 합니다.

5. 감사 로그
   - main 기준으로는 아직 예정입니다.
   - PR #19에 최근 변경 이력/감사 로그가 포함되어 있습니다.

### 운영 부족

1. Supabase 운영 DB 적용 상태표 부족
   - `schema.sql`, migrations, 실제 Supabase SQL Editor 실행 상태가 불일치할 수 있습니다.
   - 어떤 migration/seed를 어디까지 실행했는지 파일럿 DB 기준 체크가 필요합니다.

2. Production 배포 기준점 부족
   - main, PR #19, Vercel Production, Supabase DB가 서로 어떤 버전인지 원장이 테스트하기 전 명확히 고정해야 합니다.

3. 파일럿 데이터 전환 기준 부족
   - demo seed, 30명 CSV, 200명 fake seed, 실제 학원 데이터가 섞일 위험이 있습니다.
   - “시연용 DB”와 “실제 파일럿 DB”를 분리하지 않으면 원장 피드백이 흐려집니다.

4. 원장/선생님 테스트 시나리오의 실행 증거 부족
   - 문서는 많지만 실제 원장/선생님이 따라 하기 쉬운 카톡형 체크리스트와 결과 회수 양식이 더 중요합니다.

## 6. 원장 관점 부족한 부분

- 보강 불가 시간은 공유/타 학원 일정으로 표시됩니다.

부족:

- 선생님 카톡 공유를 시스템 기록으로 대체
- 학생별 히스토리 축적
- 타 학원/공유 일정으로 보강 시간 실수 방지
- 전화번호 비노출로 운영 통제

아직 약한 부분:
