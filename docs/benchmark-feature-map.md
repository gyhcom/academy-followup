# 벤치마킹 기반 기능 맵

이 문서는 해외 교육기관 SaaS의 제품 구조를 참고해 Academy Follow-up이 장기적으로 가져가야 할 기능 범위를 정리합니다.

중요한 원칙은 아래입니다.

- 화면과 메뉴는 처음부터 "문자 발송 도구"가 아니라 "학원 운영 SaaS"처럼 보이게 합니다.
- 실제 개발은 파일럿 검증에 필요한 기능부터 좁게 붙입니다.
- 기능이 많아져도 학생, 반, 선생님, 학부모 소통, 운영 리포트라는 큰 정보 구조를 유지합니다.
- 특정 레퍼런스 서비스명, 문구, 이미지, 영상 자산은 제품 안에 직접 사용하지 않습니다.

## 1. 대상 세그먼트

| 세그먼트 | 설명 | 초기 우선순위 |
| --- | --- | --- |
| 영수 전문학원 | 결석, 숙제, 재시험, 학부모 안내가 반복되는 단과형 학원 | 높음 |
| 종합학원 | 과목과 선생님이 많아 원장 운영 보드가 필요한 학원 | 높음 |
| 다지점 학원 | 캠퍼스별 기준, 템플릿, 리포트를 맞춰야 하는 조직 | 낮음 |
| 신규 개원 학원 | 처음부터 운영 체계를 잡고 싶은 학원 | 중간 |
| 전환 검토 학원 | 기존 학원관리 프로그램은 유지하고 소통/기록부터 바꾸려는 학원 | 높음 |
| 공부방/소규모 학원 | 원장이 수업과 학부모 연락을 직접 처리하는 작은 조직 | 중간 |

## 2. 제품 메뉴 구조

초기 랜딩과 앱 내부 메뉴는 아래 구조를 기준으로 확장합니다.

```text
대상
  - 영수 전문학원
  - 종합학원
  - 다지점 학원
  - 신규 개원 학원
  - 전환 검토 학원
  - 공부방/소규모 학원

기능
  - 운영 관리
    - 학생·반 관리
    - 학생 스케줄
    - 출결/결석
    - 보강/재시험
    - 선생님 업무
  - 학부모 소통
    - 문자/알림톡
    - 상담 기록
    - 공지 템플릿
    - 발송 이력
  - 성장 기능
    - 원장 대시보드
    - 수납/청구
    - 다지점 관리
    - 리포트
```

## 3. 핵심 기능 영역

### 3.1 학생·반 관리

목표:
- 모든 운영 기록의 기준점을 학생과 반으로 둡니다.

주요 기능:
- 학생 등록/수정/비활성화
- 반 등록/수정/비활성화
- 담당 선생님 지정
- 학생별 학부모 연락처 관리
- 학생별 주간 수업 스케줄
- 외부 일정/보강 불가 시간
- CSV 일괄 등록
- 휴원/퇴원 상태 관리

MVP 판단:
- 실제 파일럿을 위해 가장 먼저 필요합니다.

### 3.2 학생 스케줄/보강 관리

목표:
- 원장과 선생님이 학생별 정규 수업, 외부 일정, 보강 가능 시간을 한눈에 확인합니다.

주요 기능:
- 학생별 주간 반복 시간표
- 정규 수업/외부 일정/보강/상담 구분
- 날짜 또는 요일 선택
- 보강 후보 시간 확인
- 보강 안내 문자 생성

MVP 판단:
- 원장 요청에서 직접 나온 기능입니다. 단순 문자 앱을 넘어서는 핵심 차별점으로 우선순위를 높입니다.

### 3.3 수업 후 처리

목표:
- 수업 직후 선생님이 결석, 지각, 숙제 미완료, 재시험을 빠르게 남깁니다.

주요 기능:
- 학생별 빠른 사유 선택
- 팔로업 기록 저장
- 상태값: 초안, 발송 완료, 실패, 보류
- 담당 선생님별 처리 현황
- 보강/재시험 완료 체크

MVP 판단:
- 현재 제품의 핵심 진입 기능입니다.

### 3.4 학부모 소통

목표:
- 문자 발송을 독립 기능이 아니라 학생 기록의 일부로 만듭니다.

주요 기능:
- 문자 미리보기
- 발송 전 문구 수정
- dry-run 발송
- 실제 SMS/LMS 발송
- 카카오 알림톡
- 발송 이력
- 중복 발송 방지

MVP 판단:
- 문자 미리보기는 완료, 다음은 기록 저장과 dry-run입니다.

### 3.5 상담·학생 기록

목표:
- 상담 전 학생의 최근 흐름을 빠르게 파악합니다.

주요 기능:
- 학생 상세 페이지
- 팔로업 타임라인
- 상담 메모
- 학습 상태 메모
- 최근 결석/재시험/숙제 이력

MVP 판단:
- 팔로업 기록 저장 후 바로 붙여야 제품 가치가 커집니다.

### 3.6 원장 운영 보드

목표:
- 원장이 오늘 놓친 일을 확인하고 선생님별 처리 상태를 봅니다.

주요 기능:
- 오늘의 팔로업 수
- 미처리/완료/실패 상태
- 반별, 선생님별 집계
- 자주 발생하는 사유
- 최근 상담 필요 학생

MVP 판단:
- 친구 원장에게 보여줄 설득 화면으로 중요합니다.

### 3.7 구성원/권한

목표:
- 원장, 관리자, 선생님, 보조 선생님의 업무 범위를 분리합니다.

주요 기능:
- 선생님 등록
- 역할 지정
- 담당 반 지정
- 발송 권한 제한
- 설정 변경 권한 제한

MVP 판단:
- 원장 1명, 선생님 1명 테스트 후 확장합니다.

### 3.8 수납/청구

목표:
- 장기적으로 학원 운영 SaaS의 큰 범위를 확보합니다.

주요 기능:
- 청구 상태
- 미납 안내
- 결제 링크
- 월별 수납 리포트

MVP 판단:
- 지금 만들지 않습니다. 상용 고객 검증 후 판단합니다.

### 3.9 다지점/플랫폼 운영

목표:
- 여러 학원과 지점을 서비스가 감당할 수 있게 합니다.

주요 기능:
- 플랫폼 관리자 콘솔
- 학원 목록
- 지점 관리
- 플랜/사용량 관리
- 고객 지원 기록

MVP 판단:
- 구조와 티켓만 준비하고 개발은 뒤로 둡니다.

## 4. 개발 순서 원칙

1. 수업 후 처리에서 실제 기록을 남깁니다.
2. 학생/반/선생님 관리 기능을 붙여 seed 데이터 의존을 줄입니다.
3. 학생별 주간 스케줄로 보강 가능 시간을 확인합니다.
4. 보강 문자 생성으로 "문자 앱"이 아니라 "스케줄 기반 운영 기록"임을 보여줍니다.
5. dry-run 발송으로 비용 없이 전체 플로우를 검증합니다.
6. 원장 대시보드로 파일럿 시연 설득력을 높입니다.
7. 상용화 기능은 결제보다 온보딩, 권한, 사용량, 지원을 먼저 정리합니다.

## 5. 지금 당장 만들지 않을 것

- 학부모 앱
- 학생 앱
- 문제은행
- 온라인 강의
- 구글 캘린더 수준의 복잡한 시간표
- 수납/청구
- 다지점 고급 관리
- 자동 카카오 알림톡
- AI 기능

이 기능들은 상업화 가능성이 확인된 뒤 제품 범위에 넣습니다.
